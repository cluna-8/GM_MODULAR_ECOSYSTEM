# Operational Integration Manual: ADM_MODULAR

## 1. Token Lifecycle Management

### 1.1. How to obtain a Token
Tokens are generated by administrators. For initial deployment/testing, use the provided script:
`python seed_internal.py`
This creates a Master Token: `hcg_maestro_123`.

### 1.2. Token Format
All tokens must be sent via the standard HTTP Authorization header:
`Authorization: Bearer hcg_[token_id]`

## 2. API Operations & Routing

### 2.1. Standard Modular Endpoint
Access specialized clinical bots via:
`POST /v1/{module_id}/chat`

- `module_id` options: `chat1`, `chat2`, `chat3`, `chat4`.
- **Payload**: Standard JSON with `message` and optional `context`.

### 2.2. Legacy Tunnel (Backward Compatibility)
Existing clients using the old endpoint can continue calling:
`POST /medical/chat`
*Internally, this is routed directly to chat1.*

### 2.3. Connectivity Check
`GET /health`
Returns the status of the gateway and a list of all currently active clinical modules.

## 3. Usage & Response Monitoring

### 3.1. Consumption Tracking (Accounting)
Every response proxied through the gateway contains a `usage` summary. This data is **automatically persisted** in the `APILog` database.
- View real-time logs in the SQLite table `api_logs`.
- Monitor cumulative costs in the `tokens.total_tokens_consumed` column.

### 3.2. Error Handling Codes
- **401 Unauthorized**: Invalid `hcg_` token or missing header.
- **404 Not Found**: Attempting to call a non-existent `module_id`.
- **502 Bad Gateway**: The internal clinical module (e.g., General Chat) is down or unreachable.

## 4. Integration Best Practices
- **Security**: Never expose the `hcg_maestro_123` token in client-side code (frontend browsers). Use a backend-to-backend integration.
- **Timeouts**: Medical reasoning can be slow. Ensure your client library has a timeout of at least **60 seconds**.
